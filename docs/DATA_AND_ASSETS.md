## Item Data Merge

`ITEM_DATABASE` is generated by grouping labels under loot tags, resolving canonical IDs, applying extended details, and freezing each record. At startup, records are merged into `itemCatalog`.

Important consequences:

- Database fields can override same-named runtime fields.
- A database-only item receives generic runtime defaults.
- Runtime behavior still depends on explicit fields such as `slot`, `weaponKind`, `damage`, `healHp`, `ammoType`, and texture keys.
- Metadata such as `use.effects.hunger` does nothing until a runtime system reads it.

When adding an item, update all relevant layers and test the merged result in the browser.

## Item Naming

- Canonical IDs may preserve legacy casing for compatibility.
- Player labels should be correctly spelled and consistently title-cased.
- `ITEM_ALIASES` accepts intentional historical or display variants.
- Asset filenames use snake_case.
- Avoid creating a second item because of punctuation, capitalization, or hyphen differences.

## Inventory Icons

`ITEM_ICON_STYLE_GUIDE.md` is mandatory for new and replacement icons.

Pipeline:

1. Follow the canonical order returned by `Object.keys(ITEM_DATABASE)` for the current full-list icon pass.
2. Select approved same-family icons as references.
3. Generate one centered item or one coherent item group on a flat removable chroma background.
4. Remove chroma with soft matte and despill.
5. Crop to alpha bounds, fit within the guide's safe composition box, and center on an exact `128x128` RGBA canvas.
6. Inspect transparent corners, silhouette, wear, materials, label or pictogram readability, padding, and category scale at actual inventory size.
7. Save under a new descriptive filename until approved.
8. Attach only after approval unless integration was explicitly requested. Update the exact canonical ID in `itemCatalog` and add the file path in `itemTexturePaths`.
9. Build, load the app in a browser, inspect the console, and verify the direct asset route.
10. Regenerate Godot data when migration output is in scope. An explicit request to defer Godot work takes precedence for that pass.

Do not silently overwrite approved art.

The item-list pass is intentionally sequential: generate one candidate, wait for approval, attach it, then move to the next canonical item. As of 2026-07-17, icons through `magnet` are approved and attached. Generation is paused before `hard drive` pending further instructions.

## Sprite Sheets

Runtime sprite sheets are horizontal PNG strips. Clip definitions provide path, frame count, and frame duration.

Canonical directions:

```text
north
north_east
east
south_east
south
south_west
west
north_west
```

Source GIF naming can vary, but output filenames must use these direction keys.

Relevant tools:

- `tools/build_sprite_sheets.py`
- `tools/build_male_sprite_sheets.py`
- `tools/build_alynne_sprite_sheets.py`
- `tools/build_luis_rotation_sheets.py`
- `tools/build_ava_pickup_sheets.py`
- `tools/build_zombie_sprite_sheets.py`

Validation:

- Correct frame count and order.
- Consistent frame dimensions.
- Chroma background removed cleanly.
- Alpha retained.
- No interpolation blur.
- Direction matches the filename.
- Final action duration matches movement lock.

## Portraits

Active runtime portraits are imported through Vite at the top of `src/main.js`. Future portraits remain in `assets/portraits/` and `characterDatabase` until a sprite/animation profile makes them playable.

Portrait-processing tools:

- `tools/build_character_portraits.py`
- `tools/prepare_character_portrait.py`

Keep portrait framing consistent across inventory and Rest Station layouts.

## Environment Textures

Environment textures live in `assets/textures/`. Reuse existing maps before creating duplicates.

Patterns:

- Single-map materials use `createTextureMaterial()`.
- PBR-style props can use matching albedo, bump, and roughness files.
- Color/albedo maps use sRGB color space.
- Data maps such as bump and roughness remain linear.
- Repeating textures use `RepeatWrapping` and stable physical scale.

Relevant tools:

- `tools/generate_game_textures.py`
- `tools/generate_rest_station_assets.py`
- `tools/build_base_door_maps.py`
- `tools/build_base_fence_maps.py`
- `tools/build_crate_maps.py`

Preserve overwrite guards. Generated fallback art must not replace approved custom textures accidentally.

## Native Props and External Meshes

The current wooden supply crate is generated with lightweight Three.js geometry and shared materials. It uses:

- `crate_wood_albedo.png`
- `crate_wood_bump.png`
- `crate_wood_roughness.png`
- Existing workbench steel texture for hardware

Previously supplied Meshy GLBs were rejected because they were extremely dense and lacked game-ready UV/material data. Do not import external meshes without checking:

- Triangle and vertex count.
- UV coordinates.
- Material and texture presence.
- Static/skeleton/animation structure.
- Dimensions and axis orientation.
- Bottom-centered pivot.
- Visual quality at the actual isometric camera distance.

For small static props, native Three.js geometry can be more maintainable than an opaque generated mesh.

## Audio

Audio files live in `assets/audio/` and are imported through Vite when referenced as modules. Background tracks loop by game mode; effects are cloned from preloaded source elements.

Use short, normalized effects. Verify browser autoplay behavior through real pointer/keyboard interaction.

## House Templates

Reference images live in `assets/map_templates/house/`. Runtime definitions live in `src/data/houseMissionTemplates.js`.

A template includes:

- Unique ID and display name.
- Entrance room/side.
- Grid-positioned room records.
- Explicit connections.
- Furniture placements.
- Container and loose-loot sockets with probability.

When adding a template, verify graph connectivity, door openings, exterior entrance, furniture clearance, loot clearance, and extraction placement.

## UI Assets and Style

UI-specific art lives in `assets/ui/`. Most station styling is CSS-driven.

The learned visual language is:

- Thick charcoal/gunmetal frame.
- Layered bevels and recessed dark panels.
- Small rivets and fasteners.
- Restrained edge rust.
- Compact squared silhouettes.
- Warm off-white text and muted survival-status colors.

The downloaded reference atlas/individual PNG pack is not a runtime dependency unless direct use is explicitly requested.

## Godot Export

Run after relevant data or asset changes:

```bash
node tools/export_godot_data.mjs
```

Generated outputs include items, item textures, locations, characters, enemy types, animations, textures, upgrades, constants, and an asset manifest.

Review the JSON diff. A large asset-manifest rewrite can be legitimate after adding icons, but it should be understood before committing.

## Asset Checklist

- Correct directory and filename.
- Correct dimensions and color mode.
- Alpha where expected.
- No unintended border, background, or watermark.
- Runtime mapping updated.
- Existing art not overwritten without permission.
- Browser console has no missing-texture warning.
- Production build succeeds.
- Godot data regenerated where relevant.
